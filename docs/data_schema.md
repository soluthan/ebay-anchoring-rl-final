# Data Schema

This project treats eBay Best Offer interactions as a one-step decision problem: for each listing/thread, the buyer chooses an opening offer ratio and receives realized savings only if the offer is accepted.

No raw data or derived parquet files are committed to Git.

## Raw Inputs

Expected local source:

- `clean_master_dataset.parquet`

The source is assumed to be a merged table derived from Best Offer thread and listing data.

Important raw columns used by `data_preprocess.py`:

| Raw column | Role |
| --- | --- |
| `anon_item_id` | Anonymized listing/item identifier. |
| `anon_thread_id` | Anonymized offer-thread identifier. |
| `start_price_usd` | Seller listing price. |
| `offr_price` | Buyer's opening offer; used as the accepted price for accepted Best Offers. |
| `item_price` | Original item price field; not used as realized reward because it is approximately listing price in this dataset. |
| `status_id` | Deal status; `2` denotes an accepted offer. |
| `meta_categ_id` | High-level category; current study filters to fashion. |
| `anon_leaf_categ_id` | Anonymized leaf category. |
| `fdbk_score_src` | Seller feedback score. |
| `fdbk_pstv_src` | Seller positive-feedback percentage. |
| `src_cre_date`, `src_cre_dt`, `auct_start_dt`, `auct_end_dt` | Candidate time columns for chronological robustness splits. |

## Engineered MDP Columns

Generated by:

```bash
DATA_DIR=. OUT_DIR=./data python data_preprocess.py
```

Outputs:

- `data/train.parquet`
- `data/val.parquet`
- `data/test.parquet`
- `data/split_summary.csv`
- `data/preprocess_stats.json`

Robustness split modes:

```bash
SPLIT_MODE=random        # default random 80/10/10
SPLIT_MODE=temporal      # chronological 80/10/10
SPLIT_MODE=leaf_holdout  # disjoint anonymized fashion leaf categories
```

Seller-score normalization, seller-positive imputation, and the top leaf-category
vocabulary are fitted on the training split only and then applied to validation
and test. This avoids leaking validation/test distribution information into the
feature engineering stage.

Core columns:

| Column | Meaning |
| --- | --- |
| `log_list_price` | Log-transformed listing price. |
| `seller_score_norm` | Seller feedback score normalized by the 99th percentile. |
| `seller_pos_pct` | Seller positive-feedback percentage, median-filled when missing. |
| `categ_id_clean` | Cleaned anonymized category feature. |
| `anchor_ratio` | Buyer offer divided by listing price; the action. |
| `savings_pct` | Reward: `1 - anchor_ratio` for accepted offers, otherwise `0`. |
| `status_id` | Original acceptance status retained for evaluation. |
| `start_price_usd` | Listing price retained for dollar recommendation output. |
| `offr_price` | Buyer offer price retained for audits and dollar recommendation output; accepted-offer savings are computed as `1 - anchor_ratio`. |
| `interaction_ts` | Parsed interaction/listing timestamp used for temporal split audits. |
| `meta_categ_id_raw` | Raw high-level category retained for split diagnostics. |
| `leaf_categ_id_raw` | Raw anonymized leaf category retained for leaf-holdout diagnostics. |

Shared names and constants live in `project_constants.py`.

## Modeling Artifacts

Generated artifacts are intentionally ignored:

| Directory | Contents |
| --- | --- |
| `models/` | XGBoost models, PyTorch checkpoints, scalers, and metrics JSON files. |
| `outputs/` | Recommendation tables, dashboard figures, and OPE diagnostics. |
| `report/` | Human-facing report/presentation files and selected figures. |

## Off-Policy Evaluation Diagnostics

Generated by:

```bash
DATA_DIR=./data MODEL_DIR=./models OUTPUT_DIR=./outputs python ope.py
```

`ope.py` estimates behavior propensities because no randomized logged
propensities are available. It reports SNIPS, DR-style estimates, bootstrap
confidence intervals, effective sample size, clipped/unclipped weights, and
kernel-bandwidth sensitivity. When CQL artifacts are present, the default target
policies include `cql_offline_rl`. These outputs diagnose support mismatch and
should not be interpreted as causal live-marketplace lift.

## Caveats

- The decision process is a one-step MDP, close to a contextual bandit framing.
- PPO is trained in a learned simulator, not by interacting with the real marketplace.
- Acceptance probabilities and expected savings are model estimates. Support diagnostics should be treated as central to interpretation.
- Leaf-holdout is a within-fashion sub-category test. Cross-meta-category generalization is a stronger thesis test and remains future work.
